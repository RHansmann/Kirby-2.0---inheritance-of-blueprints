a.     Problem Description

 

Thoroughly the blueprint files of Kirby 2.0 do not support inheritance. This means that a blueprint file cannot be embedded in / merged with another blueprint file.

 

But it may be much more efficient and easier, if parameters, which are globally used, could be specified in one or more separate files and merged with each affected blueprint file instead of specifying it in each blueprint file repeatedly.

 

Modifying global parameters only once in a basic template file which is embedded in other files will be much more efficient than modifying each file separately.

 

 

 

b.     Solution


The origin Kirby file “blueprint.php” in directory “” has been modified to allow merging of blueprint files. This change has been tested and seems to work well. If you want to know how to apply these changes refer to chapter 4.

 

Please consider, that nested embedding of blueprint files is not supported, This means, that a blueprint file can be embedded in another, but embedding this again in another (third) file won’t work.

 

Nesting of embeddings makes no sense because it increases complexity. Instead – to keep a flat hierarchy of embeddings – several blueprint files can be embedded in a main file, so you can spread global fields over several files to embed.

 

 

c.     Description of Standard Parameter Definitions in Blueprint Files of Kirby

 

There are some parameters

 

-      title

-      deletable

-      tags

which can have only a single value assigned and therefore are specified in a single line:

deletable: true

 

Other parameters

-      pages

-      files

-      fields

may have more than 1 value assigned which are specified in subsequent entries (lines):

a)   Single line specifying value true for parameter pages:

pages: true

 

b)   Single line specifying value true for parameter pages:

pages: project 

 

c)   Subsequent entry for parameter “pages” (works same ways as example above):

pages:

  template: project

 

d)   Subsequent entries for parameter “pages” specifying several possible templates:

pages:

  template:

    - project

    - article

 

e)   Fields collection with subsequent fields and field “title” with subsequent parameters for this field:

fields:
  name: title

  type: text
  required: true
  label:
    en: title
    de: Titel
    es: Titulo

 

“fields” can/must have 1 or more subsequent entries describing the field(s) used. “name”, “type” and “required” are single value parameters whereas “label” can contain subsequent entries for each language.

 

d.     Description of new “merge” Parameter used in the Main Blueprint

A (template or subsequent) blueprint file which should be merged into another (main) blueprint file may contain any type of parameters allowed for the main file.

Depending on the merge parameter and its optional subsequent parameters the merging is handled in different ways depending on the type of parameterization.

Following merge parameterizations are allowed:

1.    Use filename of merged file directly in the merge line of the main file:

merge:  file2embed


This is useful if only 1 file should be merged. This type of merge command is handled in the same manner as the “end” parameter described in the following variants. If more than 1 file must be merged use following examples.

 

2.    Use “begin” and/or “end” in subsequent lines of the merge parameter to specify where new fields of the template file should be added to the fields of the main file:

merge: 

  begin: firstfile2embed

  end: secondfile2embed


     This variant allows only 1 file per merge position.

3.    To allow more than 1 file per merge position there is a third alternative for the “begin” and “end” parameter:

merge: 

  begin:

    firstfile2embed

    secondfile2embed

    ….

  end:

    thirdfile2embed

    fourthfile2embede

    ….

 

 

The following chapter describes which affect the “merge” parameter and its subsequent entries have for the merging logic.
